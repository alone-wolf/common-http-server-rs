# 问题标题
ApiResponse 的 204 快捷方法仍返回 JSON 消息体，违反 No Content 语义

# 问题摘要
204响应包含消息体

# 问题描述
当前新增了 `ApiResponse::no_content()`，其状态码为 `204 No Content`。但 `ApiResponse<T>::into_response` 会统一返回 `(status, Json(self))`，这会为 204 响应写入 JSON 消息体。按照 HTTP 语义，204 响应不应包含消息体。

# 严重程度
Medium

# 影响对象
- 模块：`src/core/response.rs`
- 接口：所有使用 `ApiResponse::no_content()` 的 HTTP 接口
- 用户范围：依赖严格 HTTP 语义或网关/代理校验的客户端

# 问题原因
`into_response` 对 `ApiResponse` 采用统一 JSON 序列化分支，没有对 `StatusCode::NO_CONTENT` 做特殊处理，导致即使声明 204 也输出了消息体。

# 核心证据路径
`src/core/response.rs`

# 待确认差异
N/A

# 造成问题的证据
- 代码路径：`src/core/response.rs:82`（`no_content`）与 `src/core/response.rs:130`（`into_response`）
- 日志/报错：N/A
- 配置位置：N/A
- 复现步骤：
  1. 在任意 handler 中返回 `ApiResponse::no_content()`。
  2. 观察响应状态码为 204。
  3. 观察响应仍包含 JSON body（例如 `{"success":true,...}`）。

# 影响
- 功能语义：HTTP 语义不一致，可能触发客户端/中间件兼容性问题。
- 维护成本：调用方以为获得“空响应”，实际仍有 body，增加排查成本。
- 发布风险：接入严格协议校验的上游网关或 SDK 时存在潜在风险。

# 建议解决方案
1. 在 `ApiResponse<T>::into_response` 中对 `StatusCode::NO_CONTENT` 增加专门分支，返回无 body 响应。
2. 保持现有其他状态码的 JSON 输出行为不变，避免影响已有 API。
3. 新增单元测试，明确验证 204 响应 body 为空。
